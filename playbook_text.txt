Dataset Selection & Strategy

PRIMARY: BETH Dataset
Source: Elastic / Kaggle — already structured like real SIEM log lines (process creation, network events, file access). Best choice for a demo because judges can immediately recognise the format as real-world.
Download: kaggle datasets download -d katehighnam/beth-dataset (approx 1.1 GB). Requires Kaggle API key.
Contains: host-level audit logs with labels for benign vs malicious activity. Perfect for showing realistic SOC alert flow.
Benign:Attack ratio is approximately 95:5 — SMOTE must be applied during training (see Phase 2).

BACKUP 1: UNSW-NB15 (~100 MB)
49 network features, clean normal vs attack labels. Small enough to iterate fast. Best if BETH download fails during setup.
Download from: research.unsw.edu.au/projects/unsw-nb15-dataset

BACKUP 2: CICIDS 2017 (~2.8 GB CSVs)
Gold standard with labelled DoS, DDoS, brute force, botnet, web attack categories. Use only if you have fast internet and time — large size can cost you 30 minutes.
Download from: unb.ca/cic/datasets/ids-2017.html

This phase builds the ingestion spine of the system. Raw SIEM logs must be parsed, normalised, and stripped of IOCs before the classifier can work. The 4x faster extraction claim comes from vectorised Pandas operations instead of loop-based regex.

Step 1.1: Log Normalisation & Schema Definition  [Rishi]
Define a standard internal alert schema as a Pydantic model: alert_id (UUID), timestamp (ISO 8601), source_ip, dest_ip, port, protocol, event_type, raw_log (original line), iocs (dict), severity_raw, label.
Write a LogParser class with a parse(raw_line: str) method. For BETH: map eventId → event_type, userId → source context, sus/evil → severity_raw. For UNSW-NB15: map srcip/dstip/proto/attack_cat directly.
Normalise all timestamps to UTC using Python's datetime module. Handle both Unix epoch integers (BETH) and human-readable strings (UNSW).
Write a batch_parse(filepath) method that reads the CSV in chunks of 10,000 rows using pd.read_csv(chunksize=10000) to avoid OOM on large files.
Output: a normalised Pandas DataFrame with the standard schema columns, ready for IOC extraction.




Step 1.2: High-Speed IOC Extraction (4x Speedup via Vectorised Pandas)  [Rishi]

IOCs to extract: IPv4 addresses, IPv6 addresses, domain names, file hashes (MD5/SHA-256), CVE IDs, URL patterns, port numbers, process names.
Compile all regex patterns into a single named-group pattern using the alternation operator |. Apply vectorised using df['raw_log'].str.extractall(compiled_pattern).
For BETH: process names and file paths are in structured columns — no regex needed. Directly assign from DataFrame columns. Reserve regex only for free-text fields.
Build an IOCExtractor class with an extract_all(df) method that returns the original DataFrame with a new iocs column containing a dict of found IOC types and values per alert.
Benchmark: time the extraction on 10,000 rows. Target is under 2 seconds. If exceeding 5 seconds, switch to concurrent.futures.ProcessPoolExecutor with max_workers=4.
Add a false_positive_filter(iocs) method that checks IPs against RFC 1918 private ranges (10.x, 172.16–31.x, 192.168.x) and strips them from external threat IOCs. Internal IPs can still be flagged as lateral movement indicators separately.




Step 1.3: Iterative Alert Verification Engine  [Rishi + Aryan]
Design a VerificationEngine class that takes a parsed alert and runs it through multiple validation passes.
Pass 1 — Structural check: Does the alert have all required fields? Is the timestamp within a valid range? Is the protocol a known value?
Pass 2 — IOC consistency check: If a destination IP is flagged as external but the port is a standard internal service port (445/SMB, 3389/RDP, 1433/MSSQL), flag as HIGH confidence lateral movement attempt.
Pass 3 — Authority-weighted knowledge base lookup: Maintain a JSON knowledge base at /data/known_bad_iocs.json containing known-bad IP ranges, CVE IDs, and known malware process names. Hits get a +30 confidence boost.
The engine runs these three passes sequentially. If any pass returns high confidence, stop and mark alert as verified. This is the iterative loop mechanism.
Output per alert: verification_status (VERIFIED/UNVERIFIED/FALSE_POSITIVE), confidence_score (0–100), evidence_trail (list of strings explaining each pass result).
The evidence_trail IS the XAI output — e.g., "alert was flagged because: IOC 185.220.x.x matches known Tor exit node range (Pass 3) + destination port 3389 indicates RDP brute force (Pass 2)".




This is where the system's intelligence core is built. The goal is a trained model that classifies alerts as BENIGN / LOW / MEDIUM / HIGH / CRITICAL and generates explainable confidence scores.

Step 2.1: Feature Engineering & Data Preparation (with SMOTE)  [Aryan]

Take the normalised DataFrame from Phase 1. Feature selection strategy differs by dataset.
For UNSW-NB15: Use these features — dur, sbytes, dbytes, sttl, dttl, sloss, dloss, sload, dload, spkts, dpkts, smeansz, dmeansz, ct_srv_src, ct_state_ttl, ct_dst_ltm, ct_src_dport_ltm, ct_dst_sport_ltm, ct_dst_src_ltm, proto (encoded), service (encoded), state (encoded).
For BETH: Use eventId, argsNum, returnValue, mountNamespace (hash-encoded), userId, parentProcessId, and engineered features like child_process_count (number of child processes spawned in a 60-second window for the same userId).
Apply LabelEncoder to categorical columns. Apply StandardScaler to numeric columns. Fit the scaler on training data ONLY — never on the test set.
Split: 70% train, 15% validation, 15% test. Use stratified split to preserve class balance.
For BETH: apply SMOTE on the training set only to balance the 95:5 skew. Import from imblearn.over_sampling.
Save the fitted scaler and encoder using joblib.dump() into /classifier/artifacts/ — required at inference time.




Step 2.2: Model Training & Selection  [Aryan]
Train three candidate models: RandomForestClassifier(n_estimators=100, n_jobs=-1), GradientBoostingClassifier(n_estimators=100), and LogisticRegression(max_iter=1000) as baseline.
Evaluate all three on the validation set using: accuracy, F1-score (macro), precision, recall, and confusion matrix via sklearn.metrics.classification_report.
The key metric for a SOC system is recall on the attack class — a missed attack is worse than a false positive. Choose the model with highest attack-class recall among those with F1 > 0.85.
RandomForest is expected to be the best performer. If training time exceeds 5 minutes, reduce n_estimators to 50.
Map output probability into 5-tier severity: P(attack) < 0.2 = BENIGN, 0.2–0.4 = LOW, 0.4–0.6 = MEDIUM, 0.6–0.8 = HIGH, > 0.8 = CRITICAL.
Save with joblib.dump(model, '/classifier/artifacts/model.pkl').
Generate feature importance: model.feature_importances_ — save as /classifier/artifacts/feature_importance.json for display in the UI.

Phase 3 adds the two features that most strongly differentiate this system. The Blast Radius Knowledge Graph quantifies business impact. The Active Deception Honeypots provide zero-false-positive detection for covered vectors.

Step 3.1: Infrastructure Knowledge Graph  [Shanteshwar]
Build a KnowledgeGraph class using NetworkX's DiGraph representing the organisation's infrastructure.
Node types: WORKSTATION, SERVER, DATABASE, DOMAIN_CONTROLLER, INTERNET_GATEWAY, HONEYPOT, CRITICAL_ASSET.
For the demo, hardcode a realistic 15–20 node topology: 1 internet gateway, 1 domain controller, 3 servers (web, app, db), 5 workstations, 1 database (critical asset), 2 honeypots, 1 backup server.
Edges represent connectivity/trust paths. Edge weights represent lateral movement difficulty (lower = easier for attacker): SMB=2, SSH=4, RDP=3, direct DB=1.
Mark DATABASE and DOMAIN_CONTROLLER nodes with critical_asset=True in their node attributes.
Persist the graph to /knowledge_graph/infra_graph.graphml using nx.write_graphml().



Step 3.2: Blast Radius Calculation with Pre-Computed Paths  [Shanteshwar]

Implement a BlastRadiusAnalyser class with a calculate(compromised_node: str) method.
For each critical_asset node, compute the shortest weighted path from the compromised node using nx.astar_path() with edge weight as heuristic cost.
Blast radius score = sum(1 / hop_distance_to_each_critical_asset). A node directly adjacent to the database scores higher than one 5 hops away.
Add temporal analysis: if the same source_ip has generated alerts on 3+ different destination nodes in the last 10 minutes, flag as LATERAL_MOVEMENT and multiply blast radius by 1.5.
Pre-compute all shortest paths at startup using nx.all_pairs_dijkstra() and store in memory as a dict. Lookups are then O(1) dict access, not live graph traversal.
Output: blast_radius_score (float), affected_nodes (list), path_to_nearest_critical (list of node names), movement_pattern (LATERAL_MOVEMENT / DIRECT_ATTACK / UNKNOWN).



Step 3.3: Active Deception Honeypot System  [Shanteshwar]
Build a HoneypotManager class that manages a registry of 5 fake assets: salary_2024_Q3_final.xlsx, db_credentials_prod.conf, /admin/login endpoint, dummy SSH on port 2222, /backup/cloud_config.json.
Each honeypot asset has: asset_id, asset_type, description, trigger_conditions, and is_triggered (bool, starts False).
The check_interaction(alert: dict) method examines raw_log, accessed_path, destination_port, and event_type to determine if it matches any honeypot trigger condition.
If a honeypot is triggered: set is_triggered=True, override severity to CRITICAL regardless of classifier output, set confidence=1.0, append "HONEYPOT TRIGGERED: 100% fidelity detection — zero false positive possible" to evidence trail, and send an immediate webhook notification.
The triggered state is intentionally impossible to unset mid-session — once a honeypot fires, that vector stays CRITICAL.




Step 4.1: LangGraph Self-Correcting Playbook FSM  [Ajaya]
Define playbook states as Python Enums: ALERT_RECEIVED, INITIAL_TRIAGE, IP_BLOCKED, ASN_BLOCKED, HOST_ISOLATED, ESCALATED_TO_L2, RESOLVED, FAILED_DEFENCE.
Implement PlaybookStateMachine using LangGraph's StateGraph. Nodes are processing functions. Edges are conditional transitions.
Transition logic: ALERT_RECEIVED → INITIAL_TRIAGE (always). INITIAL_TRIAGE → IP_BLOCKED (if severity HIGH/CRITICAL). INITIAL_TRIAGE → ESCALATED_TO_L2 (if severity MEDIUM and blast_radius > threshold). IP_BLOCKED → RESOLVED (if no further alerts from same IP within 5 minutes). IP_BLOCKED → ASN_BLOCKED (if new alert from same /24 subnet within 5 minutes — attacker switched IPs).
Implement ASN escalation: when triggering ASN_BLOCKED, extract the /16 subnet from the original IP and log the expanded block as a new playbook action.
Each state transition logs a playbook_action record: { timestamp, from_state, to_state, action_taken, reason }. Append to an actions_log list.
FAILED_DEFENCE state triggers if three consecutive remediation attempts fail. This triggers immediate L2 escalation and halts further automated action.

Step 5.1: Immutable Forensic Vault  [Ajaya]
Build a ForensicVault class that captures and seals incident evidence before any automated remediation runs.
capture_snapshot(incident: dict): takes the full incident context, serialises to JSON, computes SHA-256 hash of the serialised bytes, stores both. CRITICAL: use sort_keys=True in json.dumps to ensure deterministic key ordering.
Storage: write to /vault/snapshots/ directory. Each file: vault_{timestamp}_{alert_id}.json. Once written, do not modify — this simulates WORM storage.
verify_integrity(snapshot_id): reload the file, recompute SHA-256, compare with stored hash. Return True if match, False if tampered.
Add generate_chain_of_custody(snapshot_id) method returning: Incident ID, capture timestamp, SHA-256 hash, system hostname, captured by, and the note "This record is cryptographically sealed and legally admissible."
Write a unit test: save a snapshot, modify one byte in the file, call verify_integrity() and confirm it returns False.



Step 5.2: CACAO JSON Export  [Ajaya]

Break-Glass Protocols — If Things Break
Read this section BEFORE the hackathon, not during. These are pre-planned fallbacks for the most likely catastrophic failures.

Scenario 1: Ollama / LLM Completely Fails
Remove the LLM call from the pipeline entirely. Replace with a template-based summary generator that pulls data from the alert's fields directly.
Talking point: "Our offline LLM integration is functional but we are showing the template-based fallback to demonstrate resilience of the pipeline core."
Impact: Minimal. All other features still work. LLM is the least technically novel component.

Scenario 2: ML Model Not Training / Poor Accuracy
Switch to a rule-based classifier: if port in [3389, 445, 22] and source is external IP → HIGH. If honeypot triggered → CRITICAL. If known-bad IOC match → HIGH. Else → LOW.
Rule-based is actually easier to explain to non-technical judges: "We use a hybrid rules + ML approach."
Have this ready as RuleBasedClassifier in /classifier/rules_fallback.py. Can be coded in under 15 minutes.

Scenario 3: Dataset Download Fails Completely
Generate a synthetic dataset using numpy. Create 1,000 rows with random features. Assign labels: if source_ip starts with '185.' → attack. Enough to train and demo the classifier.
Talking point: "We designed the system to be dataset-agnostic — it connects to any SIEM log format."

Scenario 4: API Keeps Crashing
Strip out the broken module from the pipeline. The API can return partial results and still demonstrate the core classification flow.
Each module is independently importable — the API should not be a monolith that crashes entirely if one component fails.

Scenario 5: Time Is Running Out (< 2 Hours Left, Multiple Features Incomplete)
Priority 1 (non-negotiable): Classification endpoint must work. A working /classify that returns severity + evidence trail is the minimum viable demo.
Priority 2: Dashboard must show live alerts. Even a plain HTML table is enough.
Priority 3: One of — Honeypot trigger OR Blast radius (whichever is closer to done). Not both.
Drop completely if necessary: CACAO export (show the JSON file directly), LangGraph FSM (replace with simple if/else), Vault (show SHA-256 hash computed manually).
